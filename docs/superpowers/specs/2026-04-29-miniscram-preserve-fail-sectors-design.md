# miniscram preserve fail-sector pass-through — mirror redumper's descramble decision

For each data-track bin sector, mirror redumper's
`Scrambler::descramble()` decision (`cd/cd_scrambler.ixx:23-61`) when
predicting the corresponding `.scram` bytes. Today `BuildEpsilonHat`
unconditionally scrambles every non-AUDIO bin sector. For sectors
that redumper *did not* descramble (the `.fail` cases enumerated in
its `tests/unscramble/` fixtures — all-zero sectors, sectors with
invalid sync, sectors with valid sync but bad MSF/mode/data), the
bin already holds the raw scrambled bytes verbatim from disc.
Re-scrambling them produces double-scrambled garbage that doesn't
match `.scram`, forcing an entire ~2352-byte override per sector.
The fix: classify each bin sector and pass it through unchanged
when redumper would have left it scrambled.

Origin: flagged by Deterous on the VGPC (Video Game Preservation
Collective) Discord on 2026-04-29. Credit recorded in `NOTICE`.

## Goals

- Smaller delta for any disc with badly-mastered or all-zero data
  sectors. The half-life real-disc fixture (currently ~5 MB delta,
  documented as "lead-in noise + boundary sectors") is the immediate
  beneficiary; any SafeDisc-class disc that triggers the fail
  decision branches will also shrink.
- Symmetric on pack and unpack. Both paths use `BuildEpsilonHat`,
  which gets a single classifier-gated change.
- Mirror redumper's decision faithfully; don't invent a parallel
  heuristic. Pin the classifier to redumper's behaviour via their
  `tests/unscramble/` fixture set.

## Non-goals

- **Manifest schema change.** The classifier is deterministic from
  bin alone; no per-LBA list is recorded. Symmetric pack/unpack
  derive the same verdict.
- **Changes to audio handling.** `trackModeAt(...) == "AUDIO"`
  continues to short-circuit scrambling, unchanged.
- **Changes to leadin / pregap / leadout regions.** These are
  generated by `BuildEpsilonHat` from MSF math, not from bin, and
  the new classifier is bin-region-only.
- **CLI flag to disable the rule.** No knob; the rule mirrors
  redumper's behaviour and any deviation would be a bug.
- **Backwards compatibility with v1 / v2 containers.** The
  prediction algorithm changes, so override records computed
  against the old prediction can't be replayed against the new one.
  Old containers fail fast with the existing migration-error
  pattern.

## Design

### The classifier

New function in `ecma130.go`, alongside `Scramble` / `Unscramble`:

```go
// classifyBinSector mirrors redumper's Scrambler::descramble() decision
// (cd/cd_scrambler.ixx:23-61) applied to a bin sector. Returns true
// when the corresponding .scram sector at the same LBA holds
// scrambled bytes (so we must rescramble bin to predict scram), false
// when .bin == .scram for this sector (redumper passed it through;
// we do likewise).
//
// The decision tree is:
//   1. all-zero bin sector            → false (case 46: zeroed_sector)
//   2. bin[0:12] != Sync              → false (fail-type-1: invalid sync)
//   3. MSF in bin matches expected L  → true  (strong MSF check)
//   4. bin mode byte ∈ {1, 2}         → true  (sync+mode check)
//   5. bin mode byte == 0 && user-data zeroed → true (sync+mode-0-zeroed)
//   6. otherwise                      → false (fail-type-2)
//
// Only valid for data-track sectors; audio is passed through
// before this is called.
func classifyBinSector(bin []byte, expectedLBA int32) bool
```

Helper `isZeroed(buf []byte) bool` is a one-liner using `bytes.IndexFunc`
or a hand-rolled loop. The classifier is pure, no allocations.

The user-data slice in step 5 is `bin[16 : 16+2048]` — matches
redumper's `s->mode2.user_data` (the 2048-byte user-data field per
ECMA-130, immediately after the 16-byte sync+header).

### Integration in BuildEpsilonHat

`builder.go:174–176` currently reads:

```go
copy(sec[:], binBuf)
if trackModeAt(p.Tracks, lba) != "AUDIO" {
    Scramble(&sec)
}
```

The change wraps the unconditional `Scramble` call in a classifier
gate:

```go
copy(sec[:], binBuf)
if trackModeAt(p.Tracks, lba) != "AUDIO" {
    if classifyBinSector(sec[:], lba) {
        Scramble(&sec)
    }
    // else: pass through; sec already holds the (scrambled-form) bin bytes
}
```

`BuildEpsilonHat` is the single shared entry point used by both
pack (with delta encoding) and unpack (without). One change,
both directions.

### Pack reporter

The existing `building scram prediction + delta` step reports
`"%d override(s), delta %d bytes"`. Augment to track and surface
the pass-through count separately:

```
building scram prediction + delta  ✓  N overrides, M pass-throughs, delta K bytes
```

`M` is the count of bin sectors classified as fail (rescramble=false).

Plumbing: `BuildEpsilonHat`'s return tuple
`([]int32, int, error)` — currently `(errLBAs, mismatchedSectors,
err)` — gains a fourth element `passThroughs int`, incremented in
the bin-region branch every time `classifyBinSector` returns false.
The single existing call site in `pack.go::buildHatAndDelta` passes
the count up to the reporter. Purely diagnostic — no manifest field,
no behavioural effect.

### Container format version

Set the format version constant to **0** in `manifest.go` as a
sentinel. This signals the in-flight `format/drop-scrambler-hash`
branch (chunk-based binary, currently scoped to v3) that it must
rebase atop this branch and pick the real version number when
landing. Per Hugh's instruction.

`Unpack` rejects v1 and v2 containers with the existing migration-
error pattern (analogous to v0.2 → v0.3 in C1's spec).

## Error handling

- **Zero-byte bin sector that should not be zeroed.** Already
  handled today's behaviour: classifier returns false for zeroed
  bin → pass-through. If the actual `.scram` had non-zero scrambled
  bytes there, the lockstep check inside `BuildEpsilonHat` records
  an override. So a "shouldn't be zero" sector becomes a single
  override of size `SectorSize` — the same as today's behaviour for
  this case. No regression.

- **Bin sector with sync but wrong mode byte and bad MSF.** Step 6
  classifies this as fail → pass-through. The actual `.scram` is
  whatever redumper saw on disc (also pass-through), so prediction
  matches. No override needed. Improvement vs today.

- **Layout-mismatch ratio above 5%.** `LayoutMismatchError`
  continues to fire post-build at 5% threshold. With the new rule,
  legitimate disc inputs should produce *fewer* mismatches, not
  more — the threshold remains a safety net for genuinely wrong
  parameters.

## Testing

Three layers, each independently meaningful.

### 1. Redumper fixture port

Copy all 46 files from `~/redumper/tests/unscramble/` into
`testdata/unscramble/` verbatim. Add `testdata/unscramble/README.md`
crediting upstream (`superg/redumper`, GPL-3.0) and noting that
`NOTICE` already records the project-level acknowledgement.

A single test (`unscramble_fixtures_test.go`) walks the directory.
For each `<n>_<desc>.<lba>.<verdict>` file:

1. Read the scrambled sector bytes.
2. Compute the bin form by applying a Go port of redumper's
   `Scrambler::descramble()` (the test-side oracle, in
   `unscramble_oracle_test.go`):
   - If oracle returns true (descramble succeeded): bin form = the
     descrambled bytes.
   - If oracle returns false: bin form = the original scrambled
     bytes (redumper would have re-scrambled / passed through).
3. Run `classifyBinSector(binForm, lba)`.
4. Assert it returns `true` iff verdict==pass, `false` iff verdict==fail.

For files with `.null` in place of the LBA, pass
`expectedLBA = math.MinInt32`, simulating redumper's
`lba == nullptr`. `BCDMSFToLBA` cannot produce `math.MinInt32` from
any 3-byte input, so the strong-MSF branch never matches for these
fixtures, exercising the sync+mode fallback paths exclusively.

### 2. Property test

`unscramble_property_test.go` uses `testing/quick` with
`MaxCount: 1000` (or higher) to fuzz across the entire input space.
Generator produces:

- Arbitrary 2352-byte sector content.
- Arbitrary expected LBA in `[-45150, 99·60·75 - 150)`.

For each generated `(sector, lba)`:

1. Apply the test oracle (`oracleDescramble`) to produce the bin form
   and the oracle's pass/fail verdict.
2. Run `classifyBinSector(binForm, lba)`.
3. Assert `classifyBinSector == oracleDescramble.verdict`.

This catches synthesis gaps in the hand-curated fixtures.

The oracle is a Go port of `cd/cd_scrambler.ixx::descramble`,
written deliberately against redumper's source (with the same
attribution comment per CLAUDE.md's lift policy). It lives in
test code only and is *not* used by production paths — the
production classifier is what's under test.

### 3. Round-trip e2e

- Existing synthetic e2e in `e2e_test.go` and `fixtures_test.go`
  continues to assert byte-exact pack→unpack with the new rule
  applied. No change needed; synthetic discs use only "pass" sectors,
  so the new rule is a no-op for them.

- New synthetic fixture: `synthDiscWithFailSector` constructs a disc
  whose data track contains one hand-crafted "fail" sector at a
  known LBA — e.g. a sector with valid sync but mode byte 0xF7 (an
  invalid mode value). The bin holds the raw scrambled bytes; the
  scram for that LBA equals the bin (pass-through case). Pack →
  unpack → byte-equal asserted.

- `e2e_redump_test.go` half-life fixture row: bound assertions for
  delta size and error count are re-measured and updated. Both will
  shrink. Ranges are tightened by the actually-measured value;
  per-fixture comments explain the expected numbers.
